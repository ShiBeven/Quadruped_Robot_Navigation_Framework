## 阅读方法论

在看第一行代码之前，先记住三条原则：

1. **先看数据流，再看代码。** 这个项目是一个"数据管道"——LiDAR 数据从一端流入，速度指令从另一端流出。你得先把这条管道的走向刻在脑子里，再去读单个节点的实现。
2. **做减法，不做加法。** 这个项目是从一套 20+ 个包的 RoboMaster 竞赛工程精简而来的（详见设计文档）。你会发现很多"为什么没有 XXX"的疑问——答案是"被剪掉了"。
3. **仿真先行。** 不需要硬件，不需要雷达，不需要底盘，你就能跑完整条导航链路。从仿真开始，永远是从仿真开始。

### 数据流回顾

在开始阅读前，先把这个管线印在脑子里：

```
Livox LiDAR (/livox/lidar + /livox/imu)
        ↓
  point_lio (LiDAR-惯性里程计)
        ↓  aft_mapped_to_init (odom, lidar_odom 帧)
  loam_interface (帧适配 → 标准 odom 帧)
        ↓  lidar_odometry + registered_scan
  ┌─────┼─────┬────────────────┐
  ↓     ↓     ↓                ↓
sensor_scan  terrain_analysis  small_gicp_reloc
  ↓           ↓                ↓
  └─────┬─────┘                │
        ↓                      │
  Nav2 代价地图 + 规划器 + 控制器
        ↓
     cmd_vel (速度指令 → 底盘)
```

### 第 7 份：`src/navigation/nav_bringup/` 完整阅读（入口包，最重要）

**这是整个项目最重要的包，建议花最多时间在这上面。**

#### 7a. Launch 文件阅读顺序

所有 launch 文件在 `src/navigation/nav_bringup/launch/` 下。建议按依赖关系从简单到复杂阅读：

1. **`joy_teleop_launch.py`** —— 最简单的 launch，只启手柄节点。理解 launch 文件的基本结构（`Node()`、`LaunchDescription`、参数声明）。
2. **`rviz_launch.py`** —— 启动 RViz2 可视化。了解 `rviz_config` 参数和条件启动（`IfCondition`）。
3. **`navigation_launch.py`** —— **核心**。这是纯导航栈的入口。理解它启动了哪些节点：controller_server、planner_server、costmap 相关、behavior_server、terrain_analysis 各个节点、sensor_scan_generation 等。
4. **`localization_launch.py`** —— 加载地图 + 启动定位管线（point_lio + loam_interface + small_gicp），然后 include navigation_launch。
5. **`slam_launch.py`** —— SLAM 建图模式：定位管线 + slam_toolbox + navigation（无先验地图）。
6. **`simulation.launch.py`** —— 你已经读过了。回顾一下它如何组合 loopback_sim + map_server + navigation_launch。

**关键理解：** 看 `localization_launch.py` 时，注意它是如何先启动定位管线（point_lio → loam_interface → small_gicp_reloc → sensor_scan_gen → terrain_analysis），再启动导航栈的。这是一个"分层启动"的模式。

#### 7b. 配置文件阅读顺序

1. **`config/nav2_params.simulation.yaml`** —— 仿真用参数。重点关注这些段落：
   - `controller_server` → `FollowPath` —— 控制器插件是 `omni_pid_pursuit_controller::OmniPidPursuitController`，包含了 lookahead_dist、速度限制、PID 参数等。
   - `local_costmap` / `global_costmap` —— 代价地图的尺寸、分辨率、插件层（inflation_layer、obstacle_layer 等）。
   - `planner_server` —— 全局规划器是 SmacPlannerHybrid。
   - `behavior_server` —— 行为树节点（recovery 行为等）。

2. **`config/nav2_params.reality.yaml`** —— 实机参数。对比仿真版，看哪里不同（通常有更多的传感器配置、地形分析参数等）。

3. **`behavior_trees/navigate_to_pose_w_replanning_and_recovery.xml`** —— 行为树 XML。理解 Nav2 的决策流程：先规划路径 → 跟随路径 → 如果失败则执行恢复行为（旋转清理 → 后退 → 重新规划）。这是一棵"失败后会自我恢复"的树。

4. **`behavior_trees/navigate_through_poses_w_replanning_and_recovery.xml`** —— 多目标点版本，与上面的区别在于支持途经点序列。

### 第 11 份：`src/navigation/terrain_analysis/` + `terrain_analysis_ext/` — 地形分析

**两个包一起读。它们做的是同一类事情，只是范围和复杂度不同。**

1. **`terrain_analysis/src/terrainAnalysis.cpp`** —— 近场（10m）地形分析。关注：

**看完后你应该能回答：**

### 第 13 份：`src/navigation/omni_pid_pursuit_controller/` — 全向 PID 纯追踪控制器

**这是"最后一公里"——从规划好的路径到实际发给电机的速度指令。**

1. **`include/pb_omni_pid_pursuit_controller/pid.hpp`** + **`src/pid.cpp`** —— 先看 PID 控制器的纯算法实现。理解 P/I/D 三个项各自的作用。
2. **`include/pb_omni_pid_pursuit_controller/omni_pid_pursuit_controller.hpp`** —— 控制器类的接口。注意它继承自 `nav2_core::Controller`，这意味着它是 Nav2 的插件。
3. **`src/omni_pid_pursuit_controller.cpp`** —— 核心逻辑：
   - **预瞄**（Lookahead）：在路径上找前方 N 米的目标点
   - **PID 控制**：计算消除横向误差、纵向误差、航向误差所需的速度
   - **全向特性**：输出 `vx`（前后）、`vy`（横向）、`vyaw`（旋转）三个独立速度分量
4. **`pb_omni_pid_pursuit_controller.xml`** —— 插件描述文件，声明此类是 Nav2 控制器插件。
5. **`README.md`** —— 了解这个控制器的设计理念和用法。

**看完后你应该能回答：**
- "全向"是什么含义？和普通差速底盘有什么区别？

## 附录 B：新人常见困惑的提前解答

**Q: 为什么有这么多包？不能合并吗？**  
A: ROS 2 的设计哲学就是"一个包做一件事"。这种细粒度拆分意味着：换一个 LiDAR 只需要动 point_lio，换一种控制器只需要动 omni_pid_pursuit_controller，其他包不受影响。

**Q: 为什么 point_lio 和 loam_interface 是两个包而不是一个？**  
A: point_lio 是通用算法，不耦合 Nav2。loam_interface 做坐标系适配。如果你以后换了一个别的里程计算法（比如 FAST-LIO），只需要重写 loam_interface，point_lio 直接移除。

**Q: 代价地图和地形分析有什么区别？**  
A: 代价地图是 Nav2 层面的概念（"机器人规划的路径上这个格子能不能走"），地形分析是感知层面的概念（"前方 10 米的坡多陡，有没有悬垂障碍"）。先做地形分析 → 结果注入 Nav2 代价地图 → Nav2 用代价地图规划路径。

**Q: 我没有 Livox 雷达，这个项目还能用吗？**  
A: 能。point_lio 支持 Velodyne/Ouster/通用 LiDAR（看 `config/` 下的 5 个 YAML）。编译时 `--packages-skip livox_ros_driver2` 即可。

**Q: README 里说的"哨兵"是什么？**  
A: RoboMaster 机甲大师赛中的哨兵机器人——一个全向移动的自主防御机器人。本项目最早就服务于这个竞赛场景。现在已精简为通用导航框架。
